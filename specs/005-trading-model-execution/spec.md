# Feature Specification: 交易模块模型化执行与监控

**Feature Branch**: `005-trading-model-execution`  
**Created**: 2026-05-15  
**Status**: Draft  
**Input**: User description: "交易模块需要显示当前激活的模型，不同模型的交易逻辑会有区别，需要点击该模型后进行设置和是否启动，交易模块实时显示当前模型的收益情况。当前已有的双均线模型在交易模块中需要有手动触发的按钮从没有持仓的标的中筛选出买入信号的标的，和另一个功能从持仓的标的中筛选出卖出信号的标的，并且可以快速手动决策是否买入/卖出，交易模块监控是否交易成功，如果未成功需要进行撤单二次交易"

## Clarifications

### Session 2026-05-15

- Q: 扫描买入信号时，标的池是什么范围？ → A: 用户每次扫描时手动选择范围（全量/自选股/行业）
- Q: 交易模块中的模型配置面板应该放在哪里？ → A: 交易页面内联展开，不跳转到其他页面
- Q: 模型停止后重新启动时，资金池如何处理？ → A: 持久化延续，资金和持仓继续之前的状态

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 模型展示与启停管理 (Priority: P1)

用户在交易页面看到所有可用量化模型的列表，每个模型显示名称、描述、当前是否已启用。用户可以点击模型进入其配置页面，设置模型参数（如均线周期、仓位比例、时间范围），并决定是否启动该模型进行交易。启动后的模型会在后台持续运行其策略逻辑。

**Why this priority**: 模型管理是整个交易模块的入口，没有模型的选择和启停，后续的扫描、决策、下单功能都无从谈起。

**Independent Test**: 打开交易页面，验证所有活跃模型显示在列表中，可以点击模型展开/进入配置，可以修改参数并保存，点击"启动"后模型状态变为运行中。

**Acceptance Scenarios**:

1. **Given** 系统中有已注册的量化模型（如双均线模型），**When** 用户打开交易页面，**Then** 页面显示所有活跃模型列表，包含模型名称、描述和当前启用状态
2. **Given** 用户查看模型列表，**When** 用户点击某个模型，**Then** 展开该模型的配置面板，显示可编辑的参数项
3. **Given** 用户在模型配置面板中，**When** 用户修改参数并点击"启动"，**Then** 模型以新参数开始运行，状态更新为"运行中"
4. **Given** 某模型正在运行中，**When** 用户点击"停止"，**Then** 模型停止运行，不再产生新的交易信号

---

### User Story 2 - 扫描买入/卖出信号 (Priority: P1)

用户使用双均线模型时，可以通过两个手动触发按钮分别扫描市场：一个按钮从当前没有持仓的股票中筛选出出现金叉买入信号的标的，另一个按钮从当前已持仓的股票中筛选出出现死叉卖出信号的标的。扫描完成后展示结果列表，每个结果包含股票代码、名称、信号价格、信号原因等信息。

**Why this priority**: 信号扫描是交易决策的核心输入，没有信号就无法做出交易决策。这是双均线模型在交易模块中的关键工作流。

**Independent Test**: 确保有持仓数据和非持仓数据，分别点击"扫描买入信号"和"扫描卖出信号"，验证返回的结果正确（买入信号均来自非持仓股，卖出信号均来自持仓股）。

**Acceptance Scenarios**:

1. **Given** 用户已启动双均线模型，**When** 用户点击"扫描买入信号"，**Then** 系统从所有非持仓标的中筛选出满足金叉条件的股票，并以列表形式展示
2. **Given** 用户已启动双均线模型且持有部分股票，**When** 用户点击"扫描卖出信号"，**Then** 系统从当前持仓标的中筛选出满足死叉条件的股票，并以列表形式展示
3. **Given** 扫描结果列表已展示，**When** 用户查看某个信号，**Then** 可以看到该信号的股票代码、名称、触发价格、信号原因（如"MA5上穿MA20"）和信号产生时间
4. **Given** 当前没有任何满足条件的信号，**When** 用户点击扫描按钮，**Then** 系统显示"暂未发现符合条件的交易信号"

---

### User Story 3 - 手动决策与快速下单 (Priority: P1)

用户在扫描结果中，对每条信号可以快速做出决策：确认买入/卖出或忽略该信号。确认后系统立即创建对应的交易订单。用户不需要手动输入股票代码、价格、数量——系统自动根据信号信息填入默认值（价格使用信号触发价，数量根据可用资金/持仓计算）。

**Why this priority**: 快速决策是交易执行的关键环节，将信号转化为实际订单才能实现交易价值。自动化填充减少操作延迟。

**Independent Test**: 在扫描结果中点击某条买入信号的"确认买入"，验证订单自动创建且字段正确填入。

**Acceptance Scenarios**:

1. **Given** 买入信号列表中有一条信号，**When** 用户点击该信号旁的"买入"按钮，**Then** 系统自动创建买入订单（预填信号价格、计算可买数量），订单进入"已提交"状态
2. **Given** 卖出信号列表中有一条信号，**When** 用户点击该信号旁的"卖出"按钮，**Then** 系统自动创建卖出订单（预填信号价格、填入持仓数量），订单进入"已提交"状态
3. **Given** 用户看到某条信号但不想交易，**When** 用户点击"忽略"，**Then** 该信号从列表中移除并被标记为已忽略
4. **Given** 同一信号已被确认过，**When** 用户再次查看该信号，**Then** 系统显示"已确认"状态，防止重复下单

---

### User Story 4 - 订单执行监控与失败重试 (Priority: P2)

用户提交订单后，系统持续监控订单执行状态。如果订单长时间未成交，系统自动撤单并以最新市场价格重新下单。用户可以实时看到每个订单的当前状态（已提交/部分成交/全部成交/已撤单）。

**Why this priority**: 交易执行的可靠性直接影响投资收益。撤单重试机制确保交易能够最终执行，减少因流动性不足导致的错失机会。

**Independent Test**: 提交一个模拟订单，等待超时后验证系统是否自动撤单并重新下单，查看订单状态变化。

**Acceptance Scenarios**:

1. **Given** 用户已提交一个订单且超过设定时间未成交，**When** 监控周期到达，**Then** 系统自动将该订单标记为"已撤单"，并以当前最新价格创建新订单
2. **Given** 订单正在监控中，**When** 用户查看订单列表，**Then** 可以看到每个订单的实时状态和已等待时间
3. **Given** 订单已全部成交，**When** 监控检查该订单，**Then** 系统不再对该订单进行撤单重试
4. **Given** 撤单重试已达到最大次数（默认3次），**When** 订单仍未成交，**Then** 系统停止重试，将订单标记为"重试失败"并通知用户

---

### User Story 5 - 模型收益实时展示 (Priority: P2)

交易页面实时展示每个启动模型的当前收益情况，包括：累计收益率、持仓市值、可用资金、总资产曲线。数据随行情变化自动更新。

**Why this priority**: 收益展示让用户直观了解策略表现，是决策是否继续运行某模型的重要依据。P2是因为即使没有实时展示，交易本身仍可正常执行。

**Independent Test**: 启动一个模型并执行几笔交易后，验证收益面板的数据（累计收益、持仓市值等）随订单成交自动更新。

**Acceptance Scenarios**:

1. **Given** 某模型已启动并有历史交易，**When** 用户查看该模型的收益面板，**Then** 显示累计收益率、当前持仓市值、可用资金和总资产
2. **Given** 某模型正在运行且行情数据在更新，**When** 持仓股票价格变化，**Then** 持仓市值和浮动盈亏实时更新
3. **Given** 某模型从未执行过交易，**When** 用户查看收益面板，**Then** 显示初始资金状态，收益率为零

---

### Edge Cases

- 用户同时启动多个模型时，各模型的持仓和资金是否需要隔离？→ 是，每个模型独立管理资金和持仓
- 扫描信号时如果标的数量很大（全市场5000+只），性能如何处理？→ 扫描由用户手动触发，系统按批次处理并显示进度
- 订单监控在非交易时段如何处理？→ 非交易时段暂停监控，开盘后恢复
- 撤单重试时如果新价格与原订单价格偏差过大（如超过±3%），是否仍需重试？→ 超过阈值时暂停重试并提示用户手动确认
- 同一只股票同时被多个模型持有或产生信号如何处理？→ 模型间独立，各自管理各自的持仓和信号

## Requirements *(mandatory)*

### Functional Requirements

**模型管理**
- **FR-001**: 交易页面 MUST 显示所有已注册的量化模型列表，包含模型名称、描述和启用状态
- **FR-002**: 用户 MUST 能够在交易页面内点击模型卡片，以内联方式展开配置面板，编辑模型参数（如短周期、长周期、仓位比例、时间范围），无需跳转到其他页面
- **FR-003**: 用户 MUST 能够独立启动或停止每个模型，启动后模型开始运行其策略逻辑
- **FR-004**: 每个启动的模型 MUST 独立管理资金池和持仓。模型间不共享。模型停止后重新启动时，资金和持仓 MUST 延续之前的状态（不重置）

**信号扫描（双均线模型）**
- **FR-005**: 交易页面 MUST 提供"扫描买入信号"按钮，用户可选择扫描范围（全量市场/自选股/指定行业），从当前无持仓的标的中筛选满足金叉条件的股票
- **FR-006**: 交易页面 MUST 提供"扫描卖出信号"按钮，从当前已持仓的标的中筛选满足死叉条件的股票
- **FR-007**: 扫描结果 MUST 以列表形式展示，每条信号包含股票代码、名称、信号价格、信号原因和触发时间
- **FR-008**: 用户 MUST 能够对每条信号进行"确认交易"或"忽略"操作

**快速下单**
- **FR-009**: 确认买入信号时，系统 MUST 自动创建买入订单，预填信号价格并根据可用资金计算可买数量
- **FR-010**: 确认卖出信号时，系统 MUST 自动创建卖出订单，预填信号价格并填入对应持仓的全部数量
- **FR-011**: 已确认的信号 MUST 被标记为"已确认"状态，防止重复下单

**订单监控与重试**
- **FR-012**: 系统 MUST 持续监控已提交订单的执行状态（已提交 → 部分成交 → 全部成交）
- **FR-013**: 对于超过超时阈值（默认60秒）未成交的订单，系统 MUST 自动撤单并使用最新价格重新下单
- **FR-014**: 撤单重试 MUST 有最大次数限制（默认3次），超过后停止重试并提示用户
- **FR-015**: 撤单重试时如果新价格与原单偏差超过阈值（默认±3%），MUST 暂停并等待用户手动确认

**收益展示**
- **FR-016**: 每个启动的模型 MUST 展示其当前收益概况：累计收益率、持仓市值、可用资金、总资产
- **FR-017**: 模型收益数据 MUST 随行情价格变化实时更新

### Key Entities

- **模型运行实例（Model Instance）**: 一个模型在一次启动后的运行上下文，包含：关联的模型定义、当前参数配置、独立资金池、启用状态、启动时间
- **策略配置（Strategy Config）**: 模型运行时的参数配置，包含：短周期、长周期、仓位比例、时间范围、目标股票池。复用并扩展现有的 `strategy_configs` 表
- **交易订单（Order）**: 一次具体的买卖操作记录，包含：股票代码、买卖方向、价格、数量、关联信号、当前状态（已提交/部分成交/全部成交/已撤单/重试失败）、创建时间。扩展现有的 `orders` 表
- **模型持仓（Model Position）**: 某个模型实例持有的股票仓位，包含：模型标识、股票代码、持有数量、平均成本、当前市值、浮动盈亏。扩展现有的 `positions` 表以关联模型

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可以在 10 秒内完成从查看模型到启动模型的操作流程
- **SC-002**: 扫描买入/卖出信号在标的数量不超过 500 只时，结果在 5 秒内返回
- **SC-003**: 从看到信号到确认下单，用户操作不超过 3 次点击
- **SC-004**: 订单提交后 5 秒内页面显示订单状态更新
- **SC-005**: 撤单重试机制使订单最终成交率达到 95% 以上（在正常市场条件下）
- **SC-006**: 模型收益数据在行情更新后 10 秒内反映到页面

## Assumptions

- 系统运行在模拟交易环境，不涉及真实交易所接口。订单状态变化由系统内部模拟
- 每个模型使用独立的虚拟资金池（默认每模型 100,000），模型间资金隔离
- 交易时段默认为 A 股交易时间（工作日 9:30-15:00），非交易时段订单监控自动暂停
- 现有的 `strategy_configs`、`orders`、`positions` 表结构和数据将被复用和扩展
- 模型的策略逻辑（如 MA 交叉计算）复用 `backend/src/services/model_service.py` 中已有的实现
- 用户已在系统中通过"数据管理"模块下载了足够的行情数据
